1. Prompt Combinations Analysis

How many combinations of prompts are possible from Level 1 to Level 13?

Mathematical explanation:

Prompt engineering is not a fixed sequence; it is a composition of techniques. Each level represents a set of techniques. The number of combinations is the product of choices per level, assuming each level can be applied independently. However, many techniques are additive or can be mixed.

Let’s define a simplified model: each level offers a set of primary techniques:

  • Level 1: Role, Task, Constraints
  • Level 2: Context Engineering (explicit + implicit)
  • Level 3: Personas, Prime Directives
  • Level 4: Structured Input (JSON, XML, Markdown)
  • Level 5: Output structuring (length, tone, detail)
  • Level 6: Few-shot / zero-shot examples
  • Level 7: Reasoning strategies (CoT, ToT)
  • Level 8: RAG, ReAct, Reflection
  • Level 9: Prompt debugging & optimization
  • Level 10: AI Agents (single vs multi)
  • Level 11: Model selection & tuning
  • Level 12: Enterprise prompting (governance, libraries)
  • Level 13: Capstone integration

Each technique can be combined with others. If we conservatively assume 2–4 variations per level, the number of possible combinations is in the thousands. For example:

  • Role × Task × Constraints (Level 1) = 3 independent choices
  • Adding Context Engineering (Level 2) multiplies by ~3
  • Adding Persona (Level 3) multiplies by ~4
  • Structured output (Level 5) multiplies by ~3
  • Etc.

This leads to a combinatorial explosion. In practice, the conceptual combinations (distinct prompt architectures) are hundreds, but the actual instances are infinite because each prompt contains unique context and constraints.

Real use case examples of combinations:

  • BIM QA/QC Agent: Level 1 (Role=Senior QA) + Level 2 (ISO19650 context) + Level 3 (Prime Directive: never invent issues) + Level 4 (JSON input/output) + Level 6 (few-shot examples of violations) + Level 7 (CoT reasoning) + Level 8 (RAG for standards) + Level 10 (multi-agent with QA + validation) + Level 11 (Claude for reasoning, DeepSeek for extraction).
  • Revit API code generation: Level 1 (Role=Revit expert) + Level 2 (Revit version) + Level 3 (defensive programming) + Level 5 (exact output: code only) + Level 6 (few-shot with similar methods) + Level 7 (ToT for alternative approaches) + Level 9 (debugging loop).

2. Complete Learning System (Levels 1–13)

Each level includes Concept, Examples, Mini Project, Quiz, and Answers. We present a condensed but comprehensive version for each level.

📘 Level 1 – Prompt Fundamentals

Concept

The three essential components: Role (who the AI is), Task (what to do), and Constraints (how to do it). The "Golden Rule": output quality mirrors input quality.

BIM Example

Role: "Act as a Senior BIM Coordinator."
Task: "Generate a list of all sheets that have not been updated in the last 7 days."
Constraints: "Output as a bulleted list. Only include sheets from the current project phase."

Mini Project

Write a prompt for a Revit API method that creates a new floor plan view. Include role, task, and at least three constraints (e.g., transaction handling, naming convention, error logging).

Quiz

1. What are the three core components of a well-structured prompt?
2. Why is the "Golden Rule" critical for BIM automation?

Answers

1. Role, Task, Constraints.
2. Because the AI will faithfully follow your instructions; if they are ambiguous, the output will be wrong, leading to costly rework.

📘 Level 2 – Context Engineering

Concept

Provide background information (project type, Revit version, company standards) to ground the AI. Use the funnel model: broad → specific → immediate.

BIM Example

Broad: "Hospital project, ISO 19650 compliant."
Specific: "Revit 2025, Dynamo 2.19, company naming: DDDC_*."
Immediate: "Sheets in Level 01 to 05, all disciplines."

Mini Project

Write a prompt that includes a full context block for a sheet renaming script. Include project type, version, standards, and specific elements.

Quiz

1. What is the funnel model?
2. Why is explicit context better than implicit?

Answers

1. Organise context from broad to specific to immediate to avoid overwhelming the AI.
2. Implicit assumptions may be wrong; explicit context ensures the AI uses correct information.

📘 Level 3 – Prime Directives & Persona Prompting

Concept

Prime Directive: An unbreakable rule (e.g., "Never invent APIs"). Persona: A rich identity with expertise, values, style, and cognitive traits.

BIM Example

Persona: "You are a Senior BIM Automation Architect with 20 years of experience in healthcare projects. You value safety, robustness, and clarity."
Prime Directive: "Always validate inputs and use transactions with rollback."

Mini Project

Define a persona and a prime directive for a QA/QC script that checks wall parameters.

Quiz

1. What is the difference between a Role and a Persona?
2. Where should Prime Directive be placed?

Answers

1. Role is functional; Persona adds mindset, values, and cognitive style.
2. At the very top of the System message.

📘 Level 4 – Structured Prompting

Concept

Use JSON, XML, or Markdown to define input and output schemas. This reduces ambiguity and enables automation.

BIM Example

Input JSON: {"task": "rename_sheets", "new_pattern": "{original}_v{index}"}
Output JSON: {"status": "success", "renamed": [...]}

Mini Project

Design a JSON schema for a script that extracts room data (name, number, area) from Revit.

Quiz

1. Why is JSON preferred for automation?
2. Should persona be inside the JSON structure? Why?

Answers

1. It is machine-readable, parsable, and supports nested data.
2. No, because persona defines behavior, not data; keep it in system message.

📘 Level 5 – Output Structuring & Constraints

Concept

Control length (max tokens), detail level (high-level vs deep), tone (professional, urgent), and exclusions.

BIM Example

Length: "Maximum 300 words."
Detail: "Provide a high-level summary, not a detailed analysis."
Exclusion: "Do not include code snippets."

Mini Project

Write a prompt for a project status report with explicit constraints on length, structure, and tone.

Quiz

1. What is the difference between soft and hard constraints?
2. How can you enforce a word limit effectively?

Answers

1. Soft constraints are guidelines; hard constraints must be enforced (e.g., "If exceeded, the response is invalid").
2. Add a self-check rule: "Before finalising, count words and truncate if necessary."

📘 Level 6 – Few-Shot & Zero-Shot Prompting

Concept

Zero-shot: no examples; One-shot: one example; Few-shot: 3–5 examples that demonstrate the desired pattern.

BIM Example

Few-shot for clash classification:

  • Example 1: Structural column through MEP duct → Critical
  • Example 2: Architectural wall overlap with fire sprinkler → Medium
  • Example 3: Ceiling grid misaligned → Low

Mini Project

Create a few-shot prompt for a Dynamo script that creates sheets from an Excel list. Provide 3 examples with different inputs.

Quiz

1. When is few-shot better than zero-shot?
2. How many examples are optimal?

Answers

1. When the task has a specific pattern or logic that is hard to describe in words.
2. 3–5 examples, covering edge cases.

📘 Level 7 – Reasoning Strategies (CoT, ToT, Self-Consistency)

Concept

  • Chain of Thought (CoT): Step-by-step linear reasoning.
  • Tree of Thoughts (ToT): Explore multiple reasoning paths.
  • Self-Consistency: Generate multiple outputs and choose the most frequent.

BIM Example

CoT prompt for clash resolution: "Step 1: Identify all clashes. Step 2: Categorise by severity. Step 3: Suggest resolutions. Step 4: Prioritise actions."

ToT prompt: "Consider three strategies: priority-based, discipline-based, location-based. Evaluate pros/cons."

Mini Project

Write a CoT prompt for a script that validates sheet naming conventions. Include the reasoning steps.

Quiz

1. What is the difference between CoT and ToT?
2. Why is Self-Consistency valuable for code generation?

Answers

1. CoT is a single path; ToT explores multiple branches and selects the best.
2. It reduces hallucinations by verifying the output across multiple independent reasoning chains.

📘 Level 8 – Advanced Strategies (RAG, ReAct, Reflection)

Concept

  • RAG: Retrieve external documents (ISO standards, API docs) to ground the AI.
  • ReAct: Interleave reasoning with actions (tool calls).
  • Reflection: Self-critique and iterative improvement.

BIM Example

RAG: Include Revit 2025 API documentation in the prompt.

ReAct: "Fetch all sheets, check parameters, modify names, and validate."

Reflection: "Review the script for transaction handling and edge cases. If missing, fix."

Mini Project

Design a prompt that uses RAG to retrieve BIM standards and then generates a BEP section.

Quiz

1. Why is RAG critical for Revit API code generation?
2. How does Reflection improve output quality?

Answers

1. The AI may hallucinate outdated or incorrect API methods; RAG provides the exact documentation.
2. It forces the AI to identify and correct its own mistakes, leading to polished outputs.

📘 Level 9 – Prompt Debugging & Optimization

BIM Example

If a script fails on duplicate sheet names, debug: Define (output error), Examine (prompt missing duplicate handling), Build (hypothesis: add rule), Update (add rule), Gauge (test).

Mini Project

Take a prompt you have written, intentionally break it, and use the D.E.B.U.G. framework to fix it.

Answers

1. Define the symptom.
2. Because they expose weaknesses that happy-path tests miss.

3. Prompt Library System

Building an Automatic Prompt Library

A prompt library centralises reusable templates, versioning, tagging, and search.

JSON-based storage

{ "id": "revit_wall_creator_v1", "level": 7, "category": "Revit API", "tags": ["wall", "c#", "revit 2025"], "prompt": "Act as a Senior Revit API Developer...", "expected_output": "C# method that creates a wall using Transaction...", "version": "1.2", "created": "2025-01-15", "author": "BIM_Team", "usage_count": 42 }

Markdown-based system

Use front-matter in Markdown files:

--- id: revit_wall_creator level: 7 category: Revit API tags: [wall, c#] --- # Prompt Act as ...

Hybrid system (JSON + Markdown + Database)

  • Store metadata in JSON or database (PostgreSQL, MongoDB) for fast search.
  • Store the actual prompt in Markdown for readability and version control (Git).
  • Index tags, categories, and usage metrics.

Versioning

Use Git for version control, or a simple version field in JSON. Each change increments the version (e.g., v1.0, v1.1). Tag releases.

Tagging

Tags: BIM, Revit, API, Dynamo, QAQC, Python, C#, Reports, Coordination, etc. Use hierarchical tags (e.g., Revit.API, Revit.Dynamo).

Search

Implement full-text search on prompt content and metadata. Use Elasticsearch or simple SQL LIKE queries. For large libraries, use embeddings for semantic search.

Sample search query (SQL)

SELECT * FROM prompts WHERE tags @> ARRAY['Revit API'] AND level >= 5 ORDER BY usage_count DESC;

4. Local AI Agent System

Can we train a local AI agent?

Yes, you can run a local LLM (e.g., Llama 3, Mistral) using Ollama, LM Studio, or vLLM. You cannot easily train a model from scratch without massive resources, but you can fine-tune or use RAG to specialise. A local agent can perform many tasks offline with privacy.

Tools needed:

  • Model runner: Ollama, LM Studio, or llama.cpp.
  • Orchestration: LangChain or similar for agent workflows.
  • Memory: Vector database (Chroma, FAISS) for RAG and conversation memory.
  • Tool calling: Define tools (e.g., Revit API via Python) and use function calling in the model.

Architecture of a local AI agent system:

  • User Interface: Web UI or CLI.
  • Agent Core: LangChain Agent with tools, memory, and a local LLM.
  • Tool Layer: Python scripts that call Dynamo, Revit API, file I/O.
  • RAG Layer: Embed documents (BEP, standards) into a vector DB and retrieve context.
  • Memory: Short-term (conversation history) and long-term (user preferences, project data).

Memory system:

  • Short-term: Keep last N messages in context window.
  • Long-term: Store summaries in a vector DB for retrieval when relevant.

Tool calling system:

  • Define a tool with name, description, and parameters (JSON schema).
  • The agent decides to call a tool, the system executes it, and returns the result to the agent.
  • Example: get_revit_sheets(level="L01") returns a list of sheets.

RAG integration:

  • Load documents (PDF, Word, HTML) into a vector store.
  • When the user asks a question, retrieve relevant chunks and inject them into the prompt.
  • This grounds the local model in specific knowledge without fine-tuning.

Text architecture description:

The local agent consists of a LangChain agent with a local LLM (Ollama). It has access to tools for Revit operations (via pyRevit or Revit API). It also has a vector database (Chroma) containing company BIM standards. The agent plans actions, calls tools, observes results, and iterates. Memory is maintained in a conversation buffer and summarised into the vector store for long-term recall.

5. ReAct Implementation in Real Projects

What is ReAct?

ReAct = Reason + Act + Observation. The agent interleaves thinking (reasoning) with actions (tool calls) and observes the outcome to decide the next step.

How to implement in real BIM/coding projects

  1. Define the task (e.g., "Rename all sheets in Revit with a new pattern").
  2. Set up tools (e.g., Python functions to get sheets, rename sheets, log).
  3. Create a system prompt that instructs the agent to reason and act.
  4. Loop: For each step, the agent produces a thought, an action (tool call), and then observes the result.
  5. Iterate until the goal is reached or a failure occurs.

Step-by-step workflow for Revit automation

User: "Rename all sheets to include the level name prefix."

Agent Thought: "I need to fetch all sheets, get their level, and rename them."

Action: get_all_sheets() → returns list of sheets.

Observation: "Received 120 sheets."

Thought: "For each sheet, I need to read the level property and rename."

Action: rename_sheet(sheet_id, new_name) for each sheet.

Observation: "Successfully renamed 120 sheets."

Thought: "Task complete. Generate report."

Action: generate_report()

Example prompt for ReAct:

You are a BIM automation agent. You have tools: get_sheets(), rename_sheet(id, new_name), log_result(). For each task, think step by step, then choose an action. After each action, observe the result and continue. Task: Rename all sheets to include 'Level_{level}_' prefix. Begin.

7. Multi-Agent AI System (Real Life)

Concept

A multi-agent system consists of multiple AI agents, each with a specialised role. They communicate via structured messages (JSON) and collaborate to complete complex tasks.

Roles in a BIM automation pipeline

  • Planner Agent: Breaks down the user request into a sequence of tasks.
  • Code Agent: Generates code (e.g., Dynamo script, Revit API C#).
  • QA Agent: Tests the code against test cases and edge cases.
  • BIM Validation Agent: Ensures the output complies with BIM standards (ISO 19650, naming, etc.).

Communication flow

  1. User provides a high-level goal.
  2. Planner Agent creates a plan (list of subtasks).
  3. Code Agent executes each subtask, producing code snippets.
  4. QA Agent runs tests on the code; if fails, sends feedback to Code Agent.
  5. BIM Validation Agent checks the final output against standards; if issues, sends back to Code Agent.
  6. Final output is delivered to the user.

Real-world BIM automation pipeline

  • User: "Create a sheet schedule with all sheets that have revisions."
  • Planner: "Step 1: Extract sheets. Step 2: Filter by revision. Step 3: Format as CSV. Step 4: Output."
  • Code Agent: Generates Dynamo script to do the above.
  • QA Agent: Tests with sample sheets; validates output format.
  • BIM Validation: Checks sheet naming convention; if fails, requests correction.
  • Final script is delivered.

Implementation tips

  • Use a message broker or simple queue for inter-agent communication.
  • Each agent has its own system prompt and can call tools.
  • Keep communication structured (JSON) to avoid parsing errors.
