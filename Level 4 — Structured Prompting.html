📐 Level 4 — Structured Prompting

Permanent Knowledge Base • BIM Automation Specialist • Precision Through Structure

📌 Executive Summary

Structured Prompting is the practice of using formal, machine-readable formats (JSON, XML, Markdown, YAML) to organise your instructions and define the AI's output format.

The core principle is: Structured Input = Structured Output. When you provide a clearly defined schema, the AI produces a response that fits that schema perfectly—enabling direct parsing, automation, and integration with BIM tools.

Key Insight: For BIM professionals, structured prompting is the bridge between human intent and machine execution. It transforms AI from a conversation partner into an automation engine that can be safely integrated into Dynamo scripts, Revit API plugins, and project management pipelines.

🧩 Key Concepts

1. Structured Input

Definition: Using JSON, XML, or YAML to define your instructions, context, and constraints. This removes ambiguity and tells the AI exactly what information you are providing and where.

💡 Example: Instead of "We use Revit 2025," you write "revit_version": "2025". This is unambiguous and parsable.

2. Structured Output

Definition: Defining a precise schema for the AI's response. This ensures the output is complete, consistent, and ready for machine processing.

💡 Example: Including a "status": "success" field allows your automation pipeline to check the result without parsing natural language.

3. The S.C.H.E.M.A. Framework

LetterMeaningDescription
SSpecifyExact format (JSON, XML, Markdown)
CConstraintsRequired fields, types, limits
HHandleEdge cases (nulls, missing values)
EExampleProvide a sample output
MMake parsableAvoid prose inside structured data
AAsk for validationInclude a "validation" field or self-check

4. Separation of Concerns: System vs. Data

Prime Directives and Personas belong in the System message. Task data and schemas belong in the User message (often in JSON). This separation ensures that behavior rules are always enforceable and not confused with data.

🧠 Expert Mental Model

🧩 Mental Model: The Architect's Blueprint

Imagine you are an architect issuing a construction drawing. You don't write a paragraph describing a wall. You draw it to scale, with dimensions, annotations, and a legend. You use a standardised system of lines and symbols that every contractor understands.

Structured prompting is the same. You are providing the AI with a "blueprint" (the input schema) and asking for a "construction document" (the output schema) that adheres to strict industry standards.

How an Expert Thinks

  • Design for Parsing: "If I receive this JSON, can I parse it in C# in under 10 lines of code?"
  • Validate Everything: "I will include a `status` and `message` field to catch errors."
  • Separate Behavior from Data: "The Prime Directive is the *law*. The JSON is the *plan*."
  • Provide Examples: "A single example is worth 10 lines of description."

🔮 Hidden Knowledge (Rarely Written in Tutorials)

🎯 The "Status Field" Strategy

Always request a status and message field in your JSON output. This allows your automation pipeline to handle errors gracefully without human intervention.

🎯 Markdown for Instruction, JSON for Data

Use Markdown for the prose (context, persona) and JSON for the structured data (schemas, constraints). This gives the AI a clear separation between *how to think* and *what to produce*.

🎯 The "Example Output" Power Move

Including a sample output in your prompt is the single most effective way to guarantee format compliance. The AI will mimic the example structure almost exactly.

🎯 JSON Schema as Documentation

Your JSON schema becomes the *contract* between you and the AI. It documents the exact shape of data you expect, making it easy to test and validate.

📌 Things People Usually Skip (But Experts Don't)

🔸 The "Schema Validation" Request

Ask the AI to validate its own JSON against the schema before outputting. This catches errors before they reach your pipeline.

🔸 The "Null Handling" Rule

Always specify what happens if a value is missing. Should it be `null`? A default value? Skipped?

🔸 The "Token Efficiency" Mindset

Structured JSON is often more token-efficient than prose descriptions. Use it to save costs and reduce latency.

🔸 The "Integration Ready" Design

Design your output schema to match the input schema of your target system (e.g., Dynamo, Excel, Power BI). This eliminates transformation steps.

🧩 Critical Thinking Challenges

Challenge 1: The Unstructured Client

Scenario: A client sends you a messy Word document with requirements. You need to convert this into a structured prompt for the AI.

Question: How would you extract and organise the information into a JSON schema for the AI?

🔍 Expert Reasoning

Process:

  1. Identify the core task (e.g., "Generate a wall schedule").
  2. Extract key data points (e.g., wall types, parameters).
  3. Define a JSON schema: {"task": "generate_schedule", "parameters": [{"name": "string", "type": "string"}]}.
  4. Use the schema to "templatise" the client's requirements.
  5. Provide the schema to the AI, filling in the gaps with the extracted data.

This turns a messy document into a precise, actionable prompt.

Challenge 2: The Mixed Audience

Scenario: You need a report that is both human-readable and machine-parsable.

Question: How would you structure your prompt to generate both Markdown (for humans) and JSON (for machines) in a single response?

🔍 Expert Reasoning

Solution:

  • Ask for a response that contains both a Markdown section and a JSON block.
  • Use clear delimiters: "Here is the Markdown report: ..." and "Here is the JSON data: ..."
  • Define both schemas in the prompt.
  • Example: "Output Format: 1. Markdown report for review. 2. JSON array for integration."

This gives you the best of both worlds: readability and automation.

🔄 Practical Workflow: Structured Prompting

1. DEFINE THE TASK └── What is the core objective? (e.g., "Export sheet data") 2. DESIGN THE INPUT SCHEMA └── What data does the AI need? └── JSON/XML with fields, types, constraints 3. DESIGN THE OUTPUT SCHEMA └── What data should the AI return? └── Include status, message, and data fields 4. PROVIDE CONTEXT (Persona, Prime Directive) └── Place in System message, separate from schemas 5. PROVIDE AN EXAMPLE └── Show the AI exactly what a successful output looks like 6. ASK FOR VALIDATION └── "Validate your JSON against the schema before outputting."
💡 Expert Checkpoint: If your automation pipeline can parse the AI's output without human intervention, your structured prompting is production-ready.

📜 Rules to Always Remember

🏆 Golden Rules
  • Structured Input = Structured Output.
  • Separate Behavior from Data. Personas in System, JSON in User.
  • Always Include a Status Field. Enables error handling.
  • Provide an Example. It's the most effective format guide.
🚫 Never Do This
  • Put your Prime Directive inside the JSON.
  • Skip the output schema.
  • Assume the AI knows your desired structure.
  • Provide JSON without validation or example.

💼 Professional Tips: What Seniors Do Differently

  • They define schemas first. They design the output schema before writing the prompt.
  • They use validation tools. They validate the AI's JSON output against the schema automatically.
  • They keep behavior separate from data. Personas and Prime Directives never appear inside JSON.
  • They include example outputs. They know that an example is worth 10 lines of description.

⚡ Optimization Opportunities

🚀 Performance
  • JSON is more token-efficient than prose descriptions
  • Use shorter field names (e.g., "id" vs "element_id")
  • Minimise nested structures where possible
📊 Quality
  • Use JSON Schema for validation
  • Include a "self_validation" field in the schema
  • Ask the AI to verify its output against the schema
🤖 Automation
  • Build a library of common schemas (sheets, walls, parameters)
  • Automate schema validation in your pipeline
  • Generate schema from BIM standards (IFC, COBie)

🔀 Edge Cases

  • Invalid JSON output: Always include a retry mechanism in your pipeline.
  • Missing fields: Define defaults or `null` handling in the schema.
  • Nested structures: Ensure the AI can handle nesting; provide examples of nested data.
  • Large datasets: If returning large JSON, use pagination or compression (via the prompt).

✅ Validation & Quality Assurance

How to Verify Structured Output

  • Schema Validation: Use a JSON validator to check the AI's output.
  • Field Completeness: Does every required field exist?
  • Type Checking: Are strings strings? Numbers numbers?
  • Status Check: Is the `status` field "success" when expected?
  • Integration Test: Can your target system parse the output?

❓ Frequently Asked Questions

Q: Should I put my entire prompt in JSON?

A: No. Only the data should be in JSON. The Persona, Prime Directive, and contextual prose should be in natural language (often Markdown) in the System and User messages. This separation keeps behavior rules enforceable and data parsable.

Q: What if the AI returns invalid JSON?

A: Include a retry mechanism. In your pipeline, validate the JSON. If it's invalid, send the validation error back to the AI and ask it to correct the output. This is a common pattern in production AI workflows.

Q: Is Markdown considered "structured"?

A: Yes, Markdown is a structured format. It uses headings, lists, tables, and code blocks to organise information. It is less rigid than JSON but still provides predictable structure for both humans and AI.

Q: How do I handle nested data in JSON prompts?

A: Define the nested structure explicitly in the schema. Provide an example of the nested structure. The AI will mimic the nesting in its output.

Your Level 3 Expert Prompt (Without Structured Output)

Weaknesses: The output format is vague. The AI decides how to structure the response.

Production-Level with Structured Output

System: Senior BIM Architect. Never invent APIs. Output Schema: { "status": "success|failure", "code": "string (Python code)", "node_list": [ { "name": "string", "inputs": ["string"], "outputs": ["string"] } ], "errors": ["string"], "validation_passed": "boolean" } Example Output: { "status": "success", "code": "import clr...", "node_list": [...], "errors": [], "validation_passed": true }

Why it's better: The output is predictable, parseable, and includes error handling.

🏗️ Real-World Case Study

Scenario: Automating the BIM 360 Export

You need to extract all sheet data from a Revit project and export it to a CSV file that can be uploaded to BIM 360. The CSV must have specific columns: `Sheet Name`, `Sheet Number`, `Discipline`, and `Level`.

Step 1: Define the Output Schema

{ "status": "success", "data": [ { "sheet_name": "A-101", "sheet_number": "101", "discipline": "Architectural", "level": "Level 01" } ], "errors": [] }

Step 2: Write the Structured Prompt

System: Senior BIM Automation Architect. Never invent data. Task: Generate a Dynamo script that exports sheet data to CSV for BIM 360. Output Schema: (as above) Constraints: Only active sheets. Sort by sheet number. Output Format: Provide the Python code, a node list, and a sample CSV.

Step 3: Validate the Output

The AI returns JSON with the code and a sample CSV. The `status` is "success". The data is validated against the schema. The CSV is generated and uploaded to BIM 360.

Outcome

The process is fully automated. The CSV always has the correct columns, and the schema validation catches any errors before they reach BIM 360.
